TradeAlgo Product Progress Report API Internal Daily

Structured end-of-day Jira board intelligence across 7 modules — consumed by the Claude Code agent to generate daily sprint progress narratives for leadership reporting.

Overview

The Daily Jira Report is generated each day at 9:00 PM SGT. The Claude Code agent queries the Jira REST API directly against the TradeAlgo Product board on tsgs.atlassian.net, processes 7 distinct data modules, and produces a structured progress report suitable for boss-level briefings.

Agent usage: Claude Code calls the Jira API endpoints listed in this doc, transforms the responses per-module, and assembles the final narrative. Claude.ai is used for planning only — all execution and API calls happen via the agent. Pull after 9:00 PM SGT to ensure all ticket movements from the full business day are captured.

Workflow Status Order

The following is the canonical status progression for the TradeAlgo Product board. The agent uses this ordered sequence to determine whether a status transition is a forward progression, a regression, or a lateral move.

To Do→ Research→ PRD→ Mockup→ Data→ Design→ Design Approval→ Dev→ LLM→ QA→ Staging→ Prod→ GTM→ Done

Authentication

All Jira REST API calls use HTTP Basic Authentication with your Atlassian account email and a personal API token. The token must be base64-encoded as email:token.

Request Headers (all calls)

HeaderValue
AuthorizationBasic {base64("gicci.goh@attix.com:YOUR_API_TOKEN")}
Content-Typeapplication/json
Acceptapplication/json
Token generation: Create your API token at id.atlassian.com → Security → API tokens. Store it as an environment variable (JIRA_API_TOKEN) and never commit it to source control. The agent reads it from the environment at runtime.

Example (curl — base64 encoding)

# Generate your Authorization header value
echo -n "gicci.goh@attix.com:YOUR_API_TOKEN" | base64

# Use in requests
curl -s "https://tsgs.atlassian.net/rest/agile/1.0/board?name=TradeAlgo+Product" \
  -H "Authorization: Basic {YOUR_BASE64_VALUE}" \
  -H "Accept: application/json"

Before running any module, the agent resolves the TradeAlgo Product board ID. This value is cached for the session.

Agent Execution Flow

  1. Resolve board ID via board name lookup
  2. Fetch active sprint metadata
  3. Pull all sprint issues with changelogs (single paginated call)
  4. Run JQL queries for daily movement, blockers, and velocity windows
  5. Compute per-assignee velocity from issue changelog diffs
  6. Derive cycle time, scope creep, and ticket age from cached issue set
  7. Assemble all 7 module outputs into the final report JSON

Module 2 — Daily Ticket Movement

Queries all tickets updated since startOfDay() (SGT timezone) and classifies each by the type of movement detected in its changelog. New tickets and progressed tickets are reported separately.

Classification logic: The agent reads each ticket's changelog.histories filtered to today (SGT). A ticket is new if created falls within today. A ticket is progressed if its status moved forward in the canonical workflow order (To Do → Research → PRD → … → Done). Regressed if status moved backward. A ticket can appear in multiple categories if it transitioned more than once today.

Module 4 — Velocity (1D / 5D)

Counts tickets that moved forward in the workflow within each time window. New tickets raised during the window are explicitly excluded.

GET https://tsgs.atlassian.net/rest/api/3/search?jql=project = TAP AND sprint in openSprints() AND status changed AFTER "-1d" AND NOT status changed TO "To Do"&fields=status,assignee 
{
  "as_of": "2026-03-16T21:00:00+08:00",
  "window_1d": {
    "tickets_progressed": 8,
    "tickets_completed": 2      // subset of progressed that reached Done
  },
  "window_5d": {
    "tickets_progressed": 27,
    "tickets_completed": 10,
    "daily_avg_progressed": 5.4   // tickets_progressed / 5
  }
}
Exclusion rule: The JQL explicitly filters out transitions to To Do to exclude new ticket creation events. Only forward-moving transitions in the canonical workflow order count toward velocity. Regressions (e.g. QA → Dev) are excluded from both windows.

Module 5 — Assignee Velocity

Derived from the full sprint issue set (cached from Module 1). For each assignee, computes their ticket distribution across workflow stages and an overall velocity rate — defined as done / total_assigned. Statuses are bucketed into four stages for concise reporting.

{
  "as_of": "2026-03-16T21:00:00+08:00",
  "assignees": [
    {
      "assignee": "Jane Doe",
      "assigned_total": 9,
      "done": 4,
      "stage_breakdown": {
        "discovery": 1,            // To Do + Research + PRD
        "design": 2,               // Mockup + Data + Design + Design Approval
        "delivery": 2,             // Dev + LLM + QA + Staging + Prod + GTM
        "done": 4
      },
      "velocity_rate": 0.444,          // done / assigned_total — multiply by 100 for %
      "tickets_progressed_1d": 2,
      "tickets_progressed_5d": 6
    },
    {
      "assignee": "John Smith",
      "assigned_total": 7,
      "done": 2,
      "stage_breakdown": {
        "discovery": 1,
        "design": 0,
        "delivery": 4,
        "done": 2
      },
      "velocity_rate": 0.286,
      "tickets_progressed_1d": 1,
      "tickets_progressed_5d": 3
    }
    // ... one entry per assignee with tickets in this sprint
  ]
}
Stage bucketing: discovery = To Do + Research + PRD. design = Mockup + Data + Design + Design Approval. delivery = Dev + LLM + QA + Staging + Prod + GTM. Full per-status counts are available in the raw issue payload if needed.
Unassigned tickets: Tickets without an assignee are grouped under "assignee": "__unassigned__" so they are visible in the report rather than silently dropped.

Module 6 — Cycle Time

Derived entirely from the sprint issue changelog cache. For each completed ticket, measures calendar days between the first Research transition (the first active status) and the Done transition. Provides a per-issue-type breakdown.

{
  "sprint_cycle_time": {
    "avg_days": 11.4,
    "median_days": 9.5,
    "sample_size": 6              // tickets completed in current sprint
  },
  "by_issue_type": {
    "Story": { "avg_days": 13.2, "count": 3 },
    "Bug":   { "avg_days": 4.1,  "count": 2 },
    "Task":  { "avg_days": 6.8,  "count": 1 }
  },
  "by_stage_segment": {              // avg days spent in each stage group
    "discovery": 3.1,               // To Do → Design Approval
    "design": 4.2,
    "delivery": 4.1                 // Dev → GTM
  }
}
Why this matters: With a 14-stage workflow, cycle time by stage segment tells your boss exactly where tickets are spending the most time — discovery, design, or delivery. A rising delivery segment often signals dev or QA bottlenecks; a rising discovery segment points to unclear requirements upstream.

Agent timing: The agent runs at 9:00 PM SGT. This captures the full Singapore business day (9 AM–6 PM SGT) plus any after-hours updates. For sprints involving team members in other timezones, the 9 PM pull still represents the most complete snapshot available before the boss's morning review.

Jira API Pagination

Jira REST API responses are paginated. The agent handles pagination automatically using startAt and maxResults query parameters. Sprint issue calls use maxResults=100 with looping until startAt + issues.length >= total.

# Page 1
GET /rest/agile/1.0/sprint/{sprintId}/issue?startAt=0&maxResults=100

# Page 2 (if total > 100)
GET /rest/agile/1.0/sprint/{sprintId}/issue?startAt=100&maxResults=100
Rate limits: Jira Cloud enforces a rate limit of approximately 10 requests/second per user token. The agent batches calls and re-uses the sprint issue cache across all derived modules (5, 6, 7) to minimise total API calls to ~5 per report run.
